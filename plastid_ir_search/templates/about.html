Purpose

Plastid IR Search is a mini-search engine that searches for complete chloroplast (or plastid) genome records of plants and algae using the NCBI Entrez API for Python. It then compares the IDs, or Accessions, found from these searches with the records you downloaded and processed using the ir_setup command. From there, Plastid IR Search gives you a look at which of those records have inverted repeats (IRs) annotated in them using a series of color-coded symbols.

FAQs


What are plastids?


Almost all multicellular organisms have subunits, or mini-organs in their cells which we call organelles. Some of these organelles, such as mitochondria and chloroplasts, are believed to have evolved from bacteria billions of years ago and have remnants of their ancestral DNA which make up their genome.


Plastids are a group of organelles found primarily in plants, algae, and a few other multicellular organisms that take many forms. Chloroplasts, the organelles responsible for photosynthesis in these organisms, are the most well-known form of plastid.


What are inverted repeats?


Basically, these are two separate sections of the genome that are "opposites" (complements) of each other and, when lined up, match each other's opposing base. For example, IrA (the first inverted repeat) could look like:


GCTTATTCTCTATGCGGGG

and then IrB would be the same sequence backwards and have its A bases swapped with Ts and Cs swapped with Gs (and vice versa) and it will look like:
CCCCGCATAGAGAATAAGC

This file does have proper inverted repeats so why is it saying there are none found?


This program only checks if the IRs were manually annotated in the Genbank record, not if the DNA sequence truly has any inverted repeats or not. A good analogy for this is it checks if you put your name on the assignment, but not if you actually did the assignment itself.


Why did these files have to be downloaded and not instead done via the API?


NCBI states that their API can only be used at a rate of 3 request per second without an API key. If you go any faster than that, your connection could be arbitrarily closed which has already happened to me a couple of times. The rate limiting already limits searches to take around 10 seconds for 1000 results. If I were to fetch a new file from the API every time a search was executed, that would turn into 10 minutes.
